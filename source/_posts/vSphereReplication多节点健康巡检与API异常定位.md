> 本文记录一次 vSphere Replication 多节点只读巡检。主机名、地址、域名、证书内容和访问凭据均已脱敏。

## 一、巡检目标

环境包含一台管理节点和两台附加复制服务节点。巡检希望回答四个问题：

1. 三台 Appliance 是否在线；
2. 复制服务和管理服务是否正常运行；
3. 是否存在卡住的配置任务；
4. 能否从管理 API 读取配对关系、复制服务器和虚拟机复制状态。

## 二、巡检方法

本次只执行读取操作，检查以下层级：

```text
TCP/HTTPS 可达性
    ↓
Appliance 配置 API
    ↓
系统服务状态
    ↓
任务队列
    ↓
VR 管理 API 会话
    ↓
Pairings / Replication Servers / Replications
```

不能因为 Web 页面能打开，就直接判断复制链路健康。管理面、复制服务和具体复制任务是不同层级。

## 三、检查结果

三台节点的基础状态均正常：

| 检查项 | 管理节点 | 附加节点 A | 附加节点 B |
| --- | --- | --- | --- |
| HTTPS/配置接口 | 正常 | 正常 | 正常 |
| 复制服务 | Running | Running | Running |
| REST 服务 | Running | Running | Running |
| 配置任务队列 | 空 | 空 | 空 |
| VR 管理会话 | 可建立 | 不适用 | 不适用 |

这说明 Appliance 没有整体宕机，复制服务进程也在运行。

## 四、异常点

管理节点可以创建 VR API 会话，也能读取普通任务列表，但以下接口读取失败：

- 配对关系列表；
- 已注册复制服务器列表；
- 具体虚拟机复制状态。

错误指向管理网关访问本机 HMS/扩展服务时的登录、注册或证书信任问题，而不是附加复制节点离线。

同时发现部分节点的证书 Subject 与当前地址不一致，说明环境可能经历过地址变更、迁移或历史注册信息残留。

## 五、如何判断故障层级

本次结果可以分成三层：

### 5.1 已确认正常

- Appliance 在线；
- 基础配置 API 可用；
- 复制进程正在运行；
- 当前没有配置类任务堆积。

### 5.2 尚不能确认

- 哪些虚拟机正在复制；
- 当前复制进度和 RPO；
- 配对站点是否完全健康；
- 附加复制服务器是否被管理节点正确注册。

### 5.3 优先排查方向

- 管理节点到 HMS/扩展服务的认证；
- vCenter 扩展注册信息；
- 证书链和证书 Subject；
- 历史地址变更后的残留配置；
- 管理网关与复制服务之间的信任关系。

## 六、建议处理步骤

1. 在管理界面确认站点配对状态；
2. 核对 vCenter 中的 vSphere Replication 扩展注册；
3. 检查管理节点证书与当前地址是否匹配；
4. 重新建立管理节点与 HMS 的信任；
5. 确认附加复制服务器仍处于 Registered 状态；
6. 恢复 API 可见性后，再核对每台虚拟机的复制进度和 RPO；
7. 任何重新注册操作前先备份配置并安排维护窗口。

## 七、经验总结

- 服务进程 Running 不代表管理链路完整；
- 任务队列为空也不能证明虚拟机复制正常；
- 巡检应从可达性逐层深入到配对和复制对象；
- 证书、扩展注册和历史地址变更是管理 API 异常的常见原因；
- 报告必须明确区分“已确认正常”“无法确认”和“确认异常”，避免给出过度结论。
